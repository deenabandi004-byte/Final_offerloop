The Priya Desai sentence.

The "Your closest path in: Priya Desai, '22, on the Ads ML team" line is the most data-intensive thing on the Companies hero. Here's how we generate it, what data we need, and the fallback ladder when any of it is missing.

Target sentence
Your closest path in: Priya Desai, '22, on the Ads ML team. She's replied to three cohort intros this month.

01What we're actually claiming

Every clause in the sentence is a separate assertion. If any one of them is wrong, the whole thing feels generated. Breaking it down:

ClauseWhat it claimsSource
"Priya Desai"A specific real alum exists who works at {company}alumni index · current employer verified
"'22"Graduation yearalumni index · school record
"Ads ML team"Specific team, not just job titleLinkedIn enrichment · most recent title/team parse
"closest path in"Ranking — she's the best single intro candidatederived: path score (see §3)
"replied to three cohort intros"Responsive to USC students in the last ~30 daysinternal: intro tracking · messages-sent-and-responded

02Data schema required

Here's the minimum record we need on each alum to even attempt the top-tier sentence. Treat each field as independently nullable — the fallback ladder in §4 reads from whichever fields exist.

// Alumni record — what we need per person
{
  id:               "usc_priya_desai_01",
  name:             "Priya Desai",
  school:           "USC",
  grad_year:        2022,                 // null-safe
  degree:           "B.S. Data Science",
  current: {
    company_id:     "google",
    title:          "Senior ML Engineer",
    team:           "Ads ML",         // ← this is the hard one
    started_at:     "2024-03",
    verified_at:    "2026-03-11",     // stale-check
    source:         "linkedin",
    confidence:     0.92               // team parse confidence
  },
  responsiveness: {
    intros_received_30d:   5,
    intros_replied_30d:    3,
    last_replied_at:       "2026-04-18",
    reply_rate_90d:        0.71
  },
  overlap_with_user: {           // for "closest path in" ranking
    same_major:         true,
    same_minor:         false,
    shared_professors:  ["Park", "Mehta"],
    shared_clubs:       ["Trojan Data"],
    shared_internships: [],
    years_apart:        2
  },
  privacy: {
    discoverable:       true,       // alum opted in to be named
    allow_name_in_ui:   true
  }
}

The four hardest fields

  • current.team — LinkedIn rarely gives you team cleanly. You parse it out of the title, the "about" section, or recent posts. Confidence score is essential; never render a team string you're not ≥0.85 confident on.
  • current.verified_at — If the record is older than 90 days, degrade silently. "Works at Google" is safe stale; "on the Ads ML team" is not — teams change every 6–12 months.
  • responsiveness — Requires you to have tracked intros in your own system. For a new user or a company we haven't routed intros to, this will be empty. Most rows will not qualify for the top rung of the ladder in their first month of use.
  • privacy.allow_name_in_ui — Default this to false. Alumni opt in to being named in prompts like this (settings toggle on profile). Without consent, always degrade to anonymized templates. Legal + ethical.

03The "closest path in" score

One alum per company gets the "closest path in" label. We rank candidates with a small feature-weighted score and pick the top one (with a margin threshold — if #1 and #2 are tied, we say "two strong paths" instead).

// path_score(alum, user) — higher = closer path in
score = 0
// base: must be findable and consented
if !alum.privacy.allow_name_in_ui return -1

// responsiveness is the biggest signal
score += 3.0 * alum.responsiveness.reply_rate_90d
score += 1.0 * min(alum.responsiveness.intros_replied_30d, 3)

// shared context
score += 2.0 * (alum.same_major ? 1 : 0)
score += 1.0 * alum.shared_professors.length
score += 0.5 * alum.shared_clubs.length
score += 1.5 * alum.shared_internships.length

// proximity in time (recent grads have warmer ties to current students)
score += max(0, 5 - alum.overlap.years_apart) * 0.3

// freshness of our data
if stale(alum.current.verified_at, 90d) score *= 0.5

return score
The "closest path in" label only fires if the top candidate's score is ≥ 3.0 and beats the runner-up by ≥ 1.5. Otherwise degrade — it's better to say nothing than to hero a weak tie.

04The fallback ladder

Five rungs. Read in order; use the first rung whose conditions all pass. The sentence on the hero renders whatever rung matches — there is always a valid output, even for an empty alumni record.

R1
Named alum + team + responsiveness ideal
path_score ≥ 3.0 · team.confidence ≥ 0.85 · verified < 90d · privacy.allow_name_in_ui · responsiveness.intros_replied_30d ≥ 1
Your closest path in: Priya Desai, '22, on the Ads ML team. She's replied to three cohort intros this month.
The sentence the hero was designed for. Rare — probably 15–25% of company rows for an average user. That's okay; the lower rungs are still strong.
R2
Named alum + team strong
path_score ≥ 3.0 · team.confidence ≥ 0.85 · verified < 90d · privacy.allow_name_in_ui
Your closest path in: Priya Desai, '22, on the Ads ML team.
Drops the responsiveness clause. Still specific, still feels curated.
R3
Named alum, no team good
path_score ≥ 3.0 · privacy.allow_name_in_ui · (team.confidence < 0.85 OR stale)
Your closest path in: Priya Desai, '22 — same major, two shared professors.
Swaps the team for shared-context facts (major, professors, clubs, internships — whatever overlap data we do have). Feels personal without making a claim we can't defend.
R4
Anonymized, cohort stat safe
No alum meets R1–R3 threshold, but cohort data exists · count ≥ 3
Twelve USC alumni work here in data roles — three started this fall.
Same as the headline on the hero. Falls back to aggregate stats. Never names a person. This is the default most rows will land on at first.
R5
Sector-only fallback minimum
No alumni data · company is in a relevant sector for the user
A Tech · AI company on your radar — tight fit for your Data Science coursework.
Never claims Trojan presence when there isn't any. Talks about the fit instead. If even this fails, the company shouldn't be in the top 5 — demote it.

05Generation prompt (LLM-assisted sentences)

For R1–R3 we use a deterministic template. For R4–R5 we also use a template. The LLM is only used to lightly vary the phrasing for the hero detail paragraph (the 2-line prose block below the italic headline) — never to invent facts. Here's the prompt.

/* ========================================================================
   SYSTEM
   ======================================================================== */
You are Scout, a careful editorial voice inside Offerloop. You write ONE
short paragraph (2 sentences, under 40 words) for a hero card that
recommends a company to a USC student.

Rules — violating any of these is a failure:
1. Use ONLY facts from the provided JSON. Do not invent names, teams,
   numbers, dates, or relationships.
2. If a field is null or missing, you do not reference it. You never
   say "someone" or "a few people" when the data is missing.
3. Write in present tense, second person ("you," "your").
4. Tone is understated, warm, specific. No marketing voice. No emoji.
   No exclamation points. One italic clause is fine; do not overuse.
5. Keep it to 2 sentences. Target 30 words. Never exceed 40.
6. If a responsiveness number exists (intros_replied_30d), you may
   reference it in a trailing clause. Otherwise drop it silently.
7. Every fact must be defensible. If you are less than sure, omit it.

/* ========================================================================
   USER
   ======================================================================== */
Write the hero paragraph for this company, for this user, at this rung.

RUNG: R1          // which fallback tier fired
USER: {
  name: "Deena",
  school: "USC",
  major: "Data Science",
  location: "Los Angeles"
}
COMPANY: {
  name: "Google",
  sector: "Tech · AI",
  city: "San Francisco"
}
ALUMNI_SIGNAL: {
  cohort_count_in_role: 12,
  started_this_fall: 3,
  closest_path: {
    name: "Priya Desai",
    grad_year: 2022,
    team: "Ads ML",
    team_confidence: 0.92,
    intros_replied_30d: 3,
    shared: { major: true, professors: ["Park", "Mehta"] }
  }
}

/* ========================================================================
   FEW-SHOT EXAMPLES
   ======================================================================== */
Example 1 — R1 (rich)
OUT: "Your closest path in: Priya Desai, '22, on the Ads ML team.
     She's replied to three cohort intros this month."

Example 2 — R2 (no responsiveness)
OUT: "Your closest path in: Marcus Chen, '21, on the Maps infra team.
     Same major, two shared professors."

Example 3 — R3 (no team)
OUT: "Your closest path in: Amy Rodriguez, '23 — same major, Trojan Data
     alum, and she started here eight months ago."

Example 4 — R4 (anonymized)
OUT: "Twelve USC alumni work in data roles here, with three starting this
     fall. No single warmest intro yet — but the pipeline is deep."

Example 5 — R5 (sector only)
OUT: "A Tech · AI company on your radar — tight fit for your Data Science
     coursework. No tracked Trojans here yet."

/* ========================================================================
   OUTPUT FORMAT
   ======================================================================== */
Return strict JSON:
{
  paragraph: "...",
  rung_used: "R1",
  facts_used: ["name", "grad_year", "team", "intros_replied_30d"],
  facts_omitted: ["shared.professors"]
}

If you cannot write a defensible paragraph at the given rung, return:
{ paragraph: null, rung_used: "R5", reason: "insufficient data" }
and the client will degrade to the next rung down.
Why return facts_used? So the client can assert every named fact is actually present in the input JSON before rendering. If the LLM hallucinates a field, we catch it and drop to the deterministic template for that rung.

06Guardrails at render time

Even after the LLM returns clean JSON, the client re-validates before painting a single pixel.

  • Name whitelist. The only names allowed in the paragraph are alumni_signal.closest_path.name. Regex-reject any other proper noun that isn't in a static company/school whitelist.
  • Number whitelist. Every digit or spelled-out number in the paragraph must correspond to a value in facts_used. Rejects "four" when only "3" was provided.
  • Team-claim lock. If team_confidence < 0.85 and the paragraph mentions "team," strip the clause and fall back to R3.
  • Staleness stamp. Any fact pulled from a record with verified_at older than 90 days is flagged; we append "(verified in [month])" in mono 9px under the paragraph rather than silently mislead.
  • Never ship a rung without its guardrail. If validation fails twice at the same rung, drop to the next rung automatically — do not retry the LLM a third time.

07Observability

Ship this with instrumentation from day one, or you'll never know which rung is actually firing in production.

MetricTargetAlert when
R1 fire rate15–25% of hero renders< 5% for 48h — data pipeline probably broken
R5 fire rate< 20%> 40% — alumni index stale or user has thin signals
Guardrail rejection rate< 2%> 5% — prompt or model drifting
Staleness-flag rate< 10%> 25% — LinkedIn scrape cadence too slow
User "wrong!" report< 0.5%any sustained uptick — individual facts are wrong; audit sources

08What this means for design

The hero component should accept a rung prop and render accordingly. Do NOT try to render "R1 with an R4 fallback underneath" — pick one, commit, style it like it was always the only version. The italic paragraph is the same shape at every rung; only the facts inside change.

The design's job is to make every rung look intentional. If R5 feels like a sad consolation prize, we've failed. R5 should feel like Scout honestly telling you "not yet, but here's why it still matters" — which is more trustworthy than a fabricated Priya Desai.
